Overview

The idea — teach a robot to draw

G-code is the language CNC machines and drawing robots speak — a sequence of motion commands specifying where to move, at what speed, and whether the pen is up or down. Writing it by hand for anything complex is impractical. The challenge: automate that translation from any image to a valid, executable G-code file.

The project was assigned in EGR 105, Duke's introduction to engineering computation. The brief was intentionally open — build a program that generates G-code for a drawing robot. What made mine distinct was a full image processing pipeline: rather than drawing simple geometric shapes, the program analyzes real photographs or artwork, extracts meaningful contour lines, orders them intelligently to minimize pen-up travel, and writes clean G-code that the robot can execute directly.

Technical Walkthrough

How the code works

01
Image Loading & Preprocessing
The program accepts any standard image file via command-line argument. It loads it using OpenCV, converts to grayscale, and applies a Gaussian blur to suppress high-frequency noise that would otherwise produce thousands of spurious edge pixels. Blur kernel size is tunable — larger kernels produce smoother, more stylized output; smaller kernels preserve fine detail.
02
Edge Detection with Canny
The Canny edge detector identifies boundaries in the image by finding locations where pixel intensity changes sharply. It uses two thresholds — a lower and upper bound — to classify edges as strong, weak, or suppressed. I expose both thresholds as parameters so users can control how many edges get drawn: a high threshold gives a sparse, clean sketch; a low threshold produces a dense, detailed rendering.
03
Contour Extraction & Path Ordering
Raw Canny output is a pixel grid of edge points — unordered and disconnected. OpenCV's findContours groups connected edge pixels into continuous paths. The key challenge is ordering: if you draw contours in arbitrary order, the robot wastes most of its time moving between contours with the pen up. I implemented a nearest-neighbor ordering that, after finishing each contour, moves to the start of the geometrically closest one — significantly reducing total pen-up travel distance.
04
Coordinate Mapping & Scaling
Image pixels live in pixel space (top-left origin, y increasing downward). The robot works in millimeters (typically bottom-left origin, y increasing upward). The program performs a linear mapping from pixel coordinates to the robot's physical drawing area, specified by the user as width × height in mm. The aspect ratio is preserved, and the drawing is centered on the canvas automatically.
05
G-Code Generation
The final stage iterates through the ordered contour list and emits G-code commands: G0 (rapid move, pen up) to travel between contours, G1 (linear feed, pen down) to draw each stroke. A configurable feed rate controls drawing speed. The output is a plain .gcode text file that any compatible CNC controller or pen plotter can execute directly.

Code Deep Dive

Key functions

The program is structured as a pipeline of focused functions, each responsible for one transformation. Here are the two most interesting ones — contour ordering and G-code emission:

Python · robot_artist.py — contour ordering + G-code writer
import cv2
import numpy as np
from math import hypot

def order_contours_nearest_neighbor(contours):
    """Greedy nearest-neighbor ordering to minimize pen-up travel."""
    if not contours:
        return []

    ordered = []
    remaining = list(contours)
    current_pos = (0, 0)     # start at robot home

    while remaining:
        # find contour whose START point is closest to current pen position
        best_idx = min(
            range(len(remaining)),
            key=lambda i: hypot(
                remaining[i][0][0][0] - current_pos[0],
                remaining[i][0][0][1] - current_pos[1]
            )
        )
        chosen = remaining.pop(best_idx)
        ordered.append(chosen)
        # update current position to END of chosen contour
        current_pos = (chosen[-1][0][0], chosen[-1][0][1])

    return ordered


def write_gcode(contours, img_shape, canvas_mm, feed, outfile):
    """Emit G-code from ordered contour list."""
    h, w = img_shape
    cw, ch = canvas_mm          # canvas width, height in mm

    def to_mm(px, py):
        # flip Y: image origin top-left, robot origin bottom-left
        x_mm = (px / w) * cw
        y_mm = (1 - py / h) * ch
        return round(x_mm, 3), round(y_mm, 3)

    with open(outfile, 'w') as f:
        f.write("G21\nG90\nG0 Z5.0\nG0 X0 Y0\n\n")

        for i, contour in enumerate(contours):
            pts = [to_mm(pt[0][0], pt[0][1]) for pt in contour]
            f.write(f"; --- Contour {i+1} ---\n")
            f.write(f"G0 X{pts[0][0]} Y{pts[0][1]}\n")  # rapid to start
            f.write("G0 Z0.0\n")                         # pen down
            for x, y in pts[1:]:
                f.write(f"G1 X{x} Y{y} F{feed}\n")
            f.write("G0 Z5.0\n\n")                      # pen up

        f.write("G0 X0 Y0\nM2\n")                       # home + end
Design Decision
The nearest-neighbor contour ordering is a greedy approximation of the Travelling Salesman Problem — finding the shortest route that visits every contour. An exact solution is computationally intractable for hundreds of contours, but nearest-neighbor runs in O(n²) and typically cuts pen-up travel by 60–70% compared to arbitrary ordering. For a 150mm canvas, that saves several meters of wasted robot motion per drawing.

Reflection

What I learned

🧩
Decompose, Then Compose
Breaking the problem into a chain of single-responsibility functions made the code dramatically easier to debug and extend. When contour ordering wasn't working, I could test it in isolation without touching the edge detection or G-code writer. Good decomposition is the most transferable software skill I picked up.
📐
Coordinate Frames Again
The Y-axis flip between image space and robot space caused a full day of upside-down drawings before I caught it. This is essentially the same lesson from the SLAM project — coordinate frame discipline matters, always define your axes explicitly before writing a single transform.
⚡
Greedy Algorithms Are Underrated
My first instinct was to find the optimal contour ordering. Reading about TSP convinced me to try the greedy nearest-neighbor heuristic instead — it's trivial to implement and gets 80% of the benefit in 1% of the compute time. Not every problem needs an exact solution.
🎨
Physical Output Changes Everything
Running the program and getting a .gcode file felt like progress. Watching the robot actually draw the image felt like magic. The physical output made every bug fix feel worth it in a way that a purely screen-based project never quite does.
