Overview

The autonomy problem

Getting a robot to navigate autonomously in an unknown environment is one of the central problems of robotics. It requires solving two interdependent problems simultaneously: Where am I? (localization) and What does my environment look like? (mapping). SLAM is the algorithmic framework that tackles both at once.

This project started as a personal deep-dive into autonomous systems after becoming fascinated with how modern robotic platforms — from Mars rovers to autonomous vehicles — solve navigation. I built the entire system from scratch: the rover hardware platform, the sensor integration, and the full software stack.

The goal: a rover that could be placed in an unmapped indoor environment, build a map in real-time, localize itself within that map, and navigate autonomously to a sequence of goal waypoints — all without any prior map or GPS.

Build Process

From hardware to autonomy

01
Hardware Platform Design
Built a differential-drive chassis on a laser-cut acrylic frame. Key decisions: motor encoders for odometry, RPLidar A1 mounted centrally for 360° scan coverage, Raspberry Pi 4 as the compute unit, and a separate Arduino for low-level motor control (freeing the Pi for high-level processing). Designed the sensor mount geometry so the Lidar scan plane was perfectly parallel to the ground plane — critical for 2D SLAM accuracy.
02
ROS2 Ecosystem Setup
Set up ROS2 Humble on the Raspberry Pi with a cross-compilation toolchain to build packages on a laptop and deploy to the Pi. Wrote a custom URDF (Unified Robot Description Format) model defining the rover geometry, sensor frames, and joint transforms — essential for the TF2 transform tree that tells every node where each sensor is relative to the robot base.
03
Sensor Fusion with EKF
Integrated the robot_localization package for Extended Kalman Filter-based sensor fusion. The EKF fuses wheel odometry (high-frequency, prone to drift) with IMU data (good for orientation, noisy for translation) to produce a smooth, consistent odometry estimate. Tuned the process and observation noise covariances through empirical testing — this was the most time-consuming tuning phase of the project.
04
SLAM Implementation
Used SLAM Toolbox in online asynchronous mode — it processes Lidar scans as they arrive and incrementally updates the map. The key challenge was scan matching: the algorithm must correctly align successive scans to build a consistent map. Tuned the correlation parameters (search window, resolution) to balance accuracy and CPU usage on the Pi's limited compute.
05
Nav2 Autonomous Navigation
Configured Nav2 with the DWB (Dynamic Window Approach B) local planner and NavFn global planner. Set up inflation and obstacle costmaps so the robot maintains safe clearances. Implemented a waypoint-following mission node in Python that publishes navigation goals in sequence, with status monitoring to detect and recover from navigation failures.

Technical Depth

Inside the code

The waypoint mission controller is a good example of the ROS2 action client pattern — it sends goals to Nav2 asynchronously, monitors feedback, and handles failures gracefully. Below is a simplified version of the core logic:

Python · waypoint_mission.py (simplified)
class WaypointMission(Node):
    def __init__(self):
        super().__init__('waypoint_mission')
        self.client = ActionClient(self, NavigateToPose, 'navigate_to_pose')
        self.waypoints = self.load_waypoints()  # list of (x, y, yaw)
        self.current_wp = 0

    def send_next_goal(self):
        if self.current_wp >= len(self.waypoints):
            self.get_logger().info('Mission complete!')
            return

        x, y, yaw = self.waypoints[self.current_wp]
        goal = NavigateToPose.Goal()
        goal.pose = self.build_pose_stamped(x, y, yaw)

        self.client.send_goal_async(
            goal,
            feedback_callback=self.on_feedback
        ).add_done_callback(self.on_goal_accepted)

    def on_goal_accepted(self, future):
        result_future = future.result().get_result_async()
        result_future.add_done_callback(self.on_result)

    def on_result(self, future):
        status = future.result().status
        if status == GoalStatus.STATUS_SUCCEEDED:
            self.current_wp += 1
            self.send_next_goal()  # advance to next waypoint
        else:
            self.get_logger().warn('Navigation failed, retrying...')
            self.send_next_goal()  # retry same waypoint
Hardest Problem Solved
The rover kept drifting in long corridors — a classic SLAM problem called "loop closure failure". Without revisiting a known landmark, cumulative scan-matching errors caused the map to diverge. I solved this by adding pose graph optimization checkpoints: every 2 meters of travel, SLAM Toolbox attempts a loop closure against the global pose graph, snapping accumulated drift back into consistency.

Reflection

What I learned

⚖️
Hardware & Software Are Inseparable
Software problems often trace back to hardware decisions. My early localization issues were caused by wheel slippage — a mechanical problem that no amount of EKF tuning could fix. Adding encoder guards to reduce debris contact solved it. Lesson: instrument early, chase root causes, not symptoms.
🗺️
Coordinate Frames Are Sacred
I spent two days debugging a Nav2 issue that turned out to be a wrong frame ID in the costmap config. In ROS2, the entire system depends on a consistent TF tree. I now always draw the full transform tree before writing a single line of code.
🔁
Iterate in Simulation First
Halfway through, I set up Gazebo simulation so I could test algorithm changes without running the physical rover (which requires clearing a large space). This 10x accelerated my iteration cycle. I should have done it from day one.
📦
Open-Source Is a Superpower
The Nav2 and SLAM Toolbox communities were invaluable. Reading GitHub issues, understanding design decisions in source code, and contributing a small documentation fix gave me far deeper understanding than any tutorial.
